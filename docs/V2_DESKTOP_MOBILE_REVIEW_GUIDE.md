# YC Systems V2 Review Guide

This document translates the current YC Systems audit into a practical design and implementation guide for the website.

It is organized in the order the product should be evaluated and improved:

1. desktop web first
2. mobile adaptation second

The purpose is to avoid designing the desktop site one way, then improvising on mobile later.

## Core Principle

YC Systems must feel like a focused software company.

That means the site should prioritize:

- clarity
- control
- trust
- software evidence
- decision-making

It should not feel like:

- a generic agency
- a collection of unrelated pages
- a moodboard of future concepts
- a marketing site that says more than it proves

## Part 1 - Desktop Review Standard

Desktop is where the site defines hierarchy, rhythm, credibility and information architecture.

### 1. Header

Desktop header should communicate immediately:

- who we are
- what areas exist
- what the main action is

#### Rules

- use one logo treatment only
- keep the header height stable
- keep the nav vocabulary fixed
- keep one clear primary CTA
- keep the language switch light and secondary

#### Fail Conditions

- different menu labels between pages
- too many actions competing in the header
- oversized logo area that steals space from the nav
- header elements that look like separate systems

### 2. Hero

The desktop hero must explain YC Systems in one sentence.

#### Rules

- one proposition
- one support paragraph
- two main actions maximum
- real software or product evidence on the right
- Nexus as supporting layer, not the entire proof

#### Fail Conditions

- hero says several things at once
- three equal CTAs
- image is decorative instead of evidential
- abstract tech language without business outcome

### 3. Page Rhythm

Desktop pages must feel like one paced reading experience.

#### Rules

- sections must share a stable spacing rhythm
- each section should have one main purpose
- card grids should align with a common container
- text widths should remain controlled
- large headings must not dominate every section

#### Fail Conditions

- every section looks like a hero
- headings are too large for interior blocks
- cards feel randomly tall or cramped
- sections repeat the same message with different wrappers

### 4. Product Presentation

Desktop product blocks must feel like software products, not abstract concept tiles.

#### Rules

- each product gets one official status
- each card explains audience, problem and direction
- screenshots should look real and legible
- the grid should read left to right as an ecosystem

#### Fail Conditions

- conflicting labels
- repeated descriptions
- products presented without business framing
- one product visually oversized unless intentionally featured

### 5. Cases And Proof

Desktop case study sections should show proof before flourish.

#### Rules

- each case should present context and result
- screenshots should show outcome or surface quality
- live links should feel deliberate
- the case area should prove execution quality

#### Fail Conditions

- vague claims like “professional presence” with no specifics
- too much brand language and not enough evidence
- client cards that look identical to product cards without context

### 6. Method Section

The method should reduce uncertainty.

#### Rules

- six steps maximum
- one short explanation per step
- visual structure should feel operational, not inspirational
- should support the diagnosis CTA

#### Fail Conditions

- too much copy
- process looks decorative
- method duplicates what hero or footer already said

### 7. Company And Trust Pages

Desktop corporate pages should feel serious and calm.

#### Rules

- tighter typography
- more restrained visuals
- emphasis on structure, dates, contact and policy clarity
- fewer decorative effects

#### Fail Conditions

- same dramatic treatment as landing sections
- legal and trust pages reading like marketing copy
- mixed terminology between trust, privacy and company pages

## Part 2 - Mobile Review Standard

Mobile should not be a shrunk desktop. It should preserve the same logic with tighter decision flow.

### 1. Mobile Header

The mobile header should preserve identity while staying compact.

#### Rules

- logo remains readable
- menu button remains isolated and visible
- no horizontal crowding
- mobile nav must expose the same official information architecture

#### Fail Conditions

- mobile menu overlaps first content awkwardly
- CTA disappears completely
- too much padding wastes the first viewport
- the menu looks like an unrelated overlay

### 2. Mobile Hero

The mobile hero is a decision surface, not a poster.

#### Rules

- shorter line lengths
- tighter but breathable spacing
- primary CTA first
- secondary CTA second
- visual proof still visible above or near the fold

#### Fail Conditions

- headline becomes too tall and pushes everything down
- image becomes decorative and not useful
- CTAs stack awkwardly without hierarchy
- first screen ends before the user sees what YC Systems actually offers

### 3. Mobile Section Flow

Every section on mobile should answer one question and move on.

#### Rules

- section intros shorter than desktop
- cards stack with enough space but not excess height
- avoid duplicated lead paragraphs
- each block should retain action or proof

#### Fail Conditions

- long paragraph walls
- huge empty gaps between blocks
- stacked cards that feel repetitive
- too many “nice” sections with no decision value

### 4. Mobile Product Cards

Product cards must feel like product summaries, not scrolling punishment.

#### Rules

- keep image ratio stable
- keep status visible
- keep title and summary compact
- CTA should remain discoverable

#### Fail Conditions

- cards become too tall
- titles wrap badly
- images look cropped without intent
- market label or state gets buried

### 5. Mobile Cases

Cases on mobile should prioritize:

- who it was for
- what was delivered
- what the result looks like

#### Fail Conditions

- cards are too similar
- link target is unclear
- screenshots dominate but say nothing

### 6. Mobile Contact Flow

Contact must become easier on mobile, not harder.

#### Rules

- step one should be fast
- inputs should be large enough
- labels must stay visible
- success and error states must be obvious
- WhatsApp action should be useful, not chaotic

#### Fail Conditions

- too many required fields
- stacked form blocks feel endless
- error messages are small or delayed

## Shared Design Rules For Both

These rules must stay true across desktop and mobile:

### Language

- one page, one language
- no spanglish
- nav, buttons and legal terms must stay consistent

### CTA Logic

- diagnosis is the main site action
- product actions depend on product status
- cases use a case-focused action

### Visual Tone

- dark, serious, premium
- evidence over decoration
- Nexus as support, not substitute

### Exposure Control

- no repositories
- no deployment language
- no internal product risk disclosures in public copy
- no private business details

## What “Good” Looks Like

When V2 is visually and structurally correct:

- desktop feels like a serious software company homepage
- mobile feels like the same product, not a compromised version
- every page uses the same navigation language
- every page supports one business story
- the visitor always knows the next action
- the site proves more than it claims

## What We Should Improve Next

Based on the current V2 state, the next improvements should focus on:

1. unifying the shell across all canonical pages
2. correcting page titles, H1s and navigation labels
3. migrating all CTA logic to the official system
4. removing leftover old structure from interior pages
5. refining desktop rhythm before polishing mobile

Desktop defines the system.

Mobile confirms whether the system is truly disciplined.
